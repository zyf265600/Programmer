# Go 中 Slice 内存滞留与释放详解

## 1. 核心模型：Header 与底层数组

理解空间释放的前提是搞清楚 **Slice Header** ($O(1)$ 空间) 与 **底层数组** (实际内存占用者) 的关系。

```go
type slice struct {
  ptr *T	//指向底层数组
  len int	//可见长度
  cap int //从 ptr 开始的最大容量
}
```

**结论**：真正占内存的是底层数组。只要 Header 中的 `ptr` 还在引用该数组，**整个数组**在 GC 眼中就是“活跃”的，==**即便你只用了其中一个字节**==。

## 2. 内存滞留 (Memory Retention) 场景

这是 Go 开发中最经典的“伪内存泄漏”场景。

### 2.1 大数组切小片

- **现象**：从一个 100MB 的 Slice 中截取前 10 字节返回。
- **后果**：由于返回的小 Slice 引用了大数组，导致 100MB 内存全部无法回收。
- **性质**：逻辑删除 $\neq$ 物理释放。

### 2.2 缩减长度 (Reslice)

- 操作如 `s = s[1:]` 或 `s = s[:10]`。
- **==本质：仅修改 Header 的 `ptr` 和 `len`，底层数组对象依然完整存在于内存中。==**

## 3. 彻底释放内存的方案

如果你想让不再使用的底层大数组被 GC 回收，**==必须断开所有引用。==**

### 3.1 方案 A：克隆 (Clone) —— 物理搬家

通过申请一小块新内存并复制数据，断开与旧大数组的联系。

```go
// 方式 1：make + copy
small := make([]T, len(part))
copy(small, part)

// 方式 2：利用 append
small := append([]T(nil)或者[]T{}, part...)
```

### 3.2 方案 B：置空 (Nil/Empty)

- `s = nil`：Header 全清零，彻底断开引用。
- `s = s[:0]`：**注意！** 这仅清空长度，**保留**底层数组。用于**内存复用**（再次启用覆写 s 不需要重新申请内存），而非释放。

## 4. 指针切片的特殊处理 (Stale Pointers)

如果 Slice 元素是指针（如 `[]*Object`），仅修改 `len` 会导致底层数组末尾残留旧指针。

- **隐患**：底层数组没死 $\rightarrow$ 旧指针还在 $\rightarrow$ **指向的对象**无法被 GC 回收。
- **标准删除模板**：

```go
// 删除索引 i 的元素
copy(s[i:], s[i+1:])
s[len(s)-1] = nil // 显式置空，断开对对象的引用
s = s[:len(s)-1]
```

------

## 5. 决策表：复用 vs 释放

| **需求**                | **推荐操作**      | **理由**                                         |
| ----------------------- | ----------------- | ------------------------------------------------ |
| **追求性能 / 频繁操作** | `s = s[:0]`       | **避免重复分配内存，降低 GC 压力，内存占用没变** |
| **追求内存占用最小化**  | `copy` 到新 slice | 允许 GC 回收旧的大型底层数组                     |
| **彻底不再使用**        | `s = nil`         | 简单明确，直接断开引用                           |

------

## 6. 总结

- **GC 回收单位**：是整个底层数组对象，而非数组的片段。
- **Len vs Cap**：`len` 变小不代表内存占用变小；`cap` 变小也不代表旧数组已释放。
- **核心准则**：若要释放空间，必须确保没有任何 Slice Header 再指向那块底层数组。

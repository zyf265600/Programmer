# Go 中的String不可变性（Immutability）

### 1. 物理结构：石碑 vs. 活页本

不可变性的根源在于内存结构的设计。

- **String（不可变/石碑）**：底层只有 `Data` 指针和 `Len`（长度）。它**没有 `Cap`（容量）**。这意味着它在内存中是“刚刚好”的大小，没有预留任何空间供你追加。
- **Slice（可变/活页本）**：底层拥有 `Data`、`Len` 和 **`Cap`**。这个 `Cap` 就是预留的“空白页”，允许你在不搬家的情况下进行 `append`（即情况 A）。

------

### 2. 行为定义：什么是“不可变”？

在 Go 中，不可变性包含两层强制含义：

1. **禁止原地修改**：你不能通过索引修改字节，例如 `s[0] = 'H'` 会导致编译错误。底层内存被标记为只读。
2. **赋值即重定向**：当你执行 `s = "new"`，你并没有改变旧内存的内容，而是让变量 `s` 指向了一块全新的内存地址。

------

### 3. 修改字符串的“搬家”协议

因为字符串没有 `Cap` 且不可变，所有的“修改”操作（如拼接、替换）本质上都是在**复刻与重建**。

**操作步骤：**

1. **申请**：计算新长度，申请一块全新的内存。
2. **拷贝**：将旧字符串的数据完整复制到新内存。
3. **写入**：将新内容写入新内存的剩余位置。
4. **切换**：让原变量指向新地址。

> **笔记要点**：这就是为什么字符串不能直接 `append`。`append` 的精髓在于“利用剩余容量原地扩充”，而字符串根本没有多余容量。

------

### 4. 为什么需要不可变？（设计的代价与红利）

| **特性**       | **解释**                                                     |
| -------------- | ------------------------------------------------------------ |
| **并发安全**   | 多个协程（Goroutines）可以同时读取同一个字符串，无需加锁，因为谁也改不了它。 |
| **哈希稳定**   | 字符串作为 `map` 的 key 时，其哈希值永远不变，确保了查找的准确性。 |
| **零拷贝切片** | `sub := s[1:5]` 操作极其廉价。`sub` 直接复用 `s` 的底层内存，因为不用担心 `s` 会被修改导致 `sub` 跟着变。 |

------

### 5. 性能优化：如何打破不可变？

当你需要频繁修改时，必须临时打破不可变性，将“石碑”拆成“零件”：

- **`[]byte(s)` / `[]rune(s)`**：
  - **原理**：将字符串解封为可变的切片。
  - **代价**：转换瞬间会发生一次完整的内存拷贝。
- **`strings.Builder`**：
  - **原理**：内部维护一个 `[]byte`（拥有 `Cap`），把多次“搬家”合并为一次或极少数次。
  - **场景**：大规模字符串拼接（如在循环中追加内容）。

------

### 💡 核心金句（记在页边）：

> **不可变性让“读取”变得极度廉价和安全，但让“修改”变得昂贵。**
>
> **当你感到修改字符串很麻烦时，那正是 Go 在提醒你注意内存分配的成本。**
